# Skill Studio Pro 系统需求规格（SPEC）

文档状态：Draft 0.2（V1 产品闭环修订）
日期：2026-07-16
产品名称：Skill Studio Pro  
上游基线：`liu673/skill-studio` commit `cd0bb0af53865d4a9643968080bfc5a8137b72d9`

## 1. 文档目的

本文件定义 Skill Studio Pro 第一代开源桌面产品必须满足的可验证行为。PRD 负责补充用户价值和交互流程，技术方案负责定义实现方式，自动化测试文档负责将本文件中的需求映射为测试。

## 2. 产品定义

Skill Studio Pro 是一个本地优先、跨平台、多 Agent 的 Skill 资产管理桌面应用。它自动发现电脑中分散在不同 AI Agent、插件和项目目录中的 Skill，统一展示 Skill 的用途、用法、原文、安装位置和来源，并提供中央库、多 Agent 映射、安装、编辑、同步、回收站、AI 信息整理与来源可信度功能。

## 3. 术语

| 术语 | 定义 |
|---|---|
| Skill | 以 `SKILL.md` 为核心，可包含脚本、参考资料、资源和元数据的能力目录 |
| 外部实例 | 扫描到的、当前位于某个 Agent、插件或项目目录中的 Skill 实体 |
| 中央 Skill | 由 Skill Studio Pro 工作区管理的 Skill 主副本 |
| 映射 | 中央 Skill 在某个 Agent 目录中的受管副本或符号链接 |
| 来源 | Skill 的官方仓库、Git 仓库、本地路径、插件、市场或人工创建记录 |
| 来源可信度 | 系统根据可验证证据计算的 0–100 整数，以及对应判断依据 |
| 纳管 | 将一个外部实例复制到中央库并建立关联，原文件默认不移动 |
| 发布/同步 | 将中央 Skill 的指定快照写入一个或多个 Agent Skill 目录 |
| 回收站 | 应用控制的可恢复删除区，不等同于立即永久删除 |
| AI 采集 | 使用 MiniMax 对 Skill 内容进行字段提取、分类和标签整理 |
| AI 提炼 | 使用 OpenAI GPT-5.6 对内容进行简介生成、冲突判断和结果提炼 |

## 4. 目标与非目标

### 4.1 产品目标

1. 用户无需查找隐藏目录即可看到本机 Skill 全貌。
2. 用户可以判断每个 Skill 的用途、位置、来源、版本和风险。
3. 用户可以用一个中央主副本服务多个 Agent。
4. 用户可以安全地安装、编辑、同步、移除、删除和恢复 Skill。
5. AI 生成内容必须可追溯到模型、职责和时间。
6. 产品在 Windows、macOS、Linux 上保持同一数据模型和核心行为。
7. 产品完全开源并保留上游归属与许可证义务。

### 4.2 第一代产品非目标

1. 不提供用户账号、登录或在线身份系统。
2. 不提供云同步、云备份或远程托管 Skill 库。
3. 不提供团队空间、审批流和多人协作。
4. 不自动执行 Skill 目录中的脚本。
5. 不承诺自动判断 Skill 是否绝对安全。
6. 不将 AI 推断结果冒充为可验证事实。
7. 不把 Apple 的专有设计资源直接打包到跨平台产品中。

## 5. 总体原则

1. **复用优先**：优先复用上游 Skill Studio 的 UI、Tauri 命令、SQLite、导入、快照、平台识别和同步能力。
2. **本地确定性优先**：路径、文件、Git、manifest、哈希和 YAML 等可由代码确定的信息不得交给模型猜测。
3. **非破坏性优先**：首次扫描不移动、不覆盖、不删除现有 Skill。
4. **可恢复优先**：编辑、覆盖、发布和删除前必须具备恢复路径。
5. **证据可见**：来源和可信度必须显示判断依据。
6. **模型透明**：AI 结果必须显示服务商、模型、任务职责和生成时间。
7. **密钥不落明文**：API Key 不得写入 SQLite、普通 JSON、日志或 Git。

## 6. 功能需求

### 6.1 本机发现与盘点

#### INV-001 自动平台发现

应用必须识别并允许配置至少以下 Agent 的全局 Skill 目录：

- Codex
- Claude Code
- Cursor
- Windsurf
- Gemini CLI

应用应继续复用上游已提供的 40+ 平台定义。未内置的平台必须支持用户添加自定义平台名称和 Skill 目录。

验收标准：在测试主目录放置五类标准目录后，扫描结果正确标记平台、绝对路径和检测状态。

#### INV-002 多根目录扫描

应用必须扫描：

- Agent 全局 Skill 目录
- 用户配置的额外目录
- 已知插件缓存或插件 Skill 目录
- 用户显式加入的项目级目录
- 中央 Skill 库

扫描必须支持嵌套目录，但只有存在可读取 `SKILL.md` 的目录才能登记为 Skill 实例。

验收标准：混合普通文件夹、嵌套 `.system` 和插件目录的测试树中，只登记有效 Skill 目录。

#### INV-003 非破坏性首次扫描

首次扫描只能读取文件和写入应用索引。不得移动、重命名、覆盖、删除或修改被扫描目录中的内容。

验收标准：扫描前后被扫描目录的文件树、内容哈希和修改时间保持不变。

#### INV-004 增量扫描

应用必须保存上次扫描状态，并根据路径、修改时间、大小和内容哈希识别新增、修改、移动和消失的实例。用户必须可以手动执行强制全量扫描。

验收标准：只修改一个 `SKILL.md` 时，增量扫描只更新受影响实例及其派生信息。

#### INV-005 文件监听

应用运行期间应监听已启用扫描根目录的变化。监听不可用、权限不足或事件丢失时，必须显示状态并允许手动重扫。

#### INV-006 元数据解析

系统必须解析：

- YAML Front Matter 中的 `name`、`description`、`metadata`
- `metadata.short-description`
- Markdown 标题和常见章节
- Skill 文件树
- 脚本、引用、资源和可执行文件存在性
- Git remote、当前 commit 和工作区状态（如果存在）
- 插件 manifest 和市场元数据（如果存在）

YAML 无效、UTF-8 BOM 或编码异常不得导致整次扫描失败，必须形成单 Skill 错误记录。

#### INV-007 扫描根管理界面

设置或平台中心必须提供真实的扫描根管理能力。用户必须能查看根类型、平台、绝对路径、可用性、启用状态、监听状态和最近扫描时间，并能添加、编辑、停用或移除自定义扫描根。

移除扫描根只删除应用索引配置，不得删除根目录或其中的 Skill。路径测试、保存和扫描失败必须显示可操作错误；用户可以只扫描选定根，也可以强制全量扫描。

### 6.2 Skill 列表、详情与检索

#### CAT-001 Skill 列表

列表至少显示：名称、简短说明、平台、中央库状态、来源类型、可信度、风险标记、最近更新时间和重复状态。

#### CAT-002 搜索与筛选

用户必须能按名称、描述、标签、平台、来源、可信度区间、纳管状态、脚本存在性、错误状态和更新时间搜索或筛选。

#### CAT-003 Skill 详情

详情页必须显示：

- 名称及全部路径实例
- 本地解析信息
- AI 简短说明
- 使用建议
- 原始 `SKILL.md` 快速查看入口
- 完整文件树
- 来源、可信度和判断依据
- Git、插件和安装记录
- 已映射 Agent
- 快照和操作历史
- 脚本、外部依赖和环境变量提示
- AI 结果的模型归属信息

#### CAT-004 原文查看

用户必须能在两次点击以内打开 `SKILL.md` 原文。原文视图必须明确区分“只读预览”和“编辑模式”。

#### CAT-005 重复与冲突

系统必须按标准化名称、目录名和内容哈希识别：

- 同名同内容
- 同名不同内容
- 同内容不同名称
- 中央库与 Agent 映射漂移

系统不得自动合并冲突项。

### 6.3 来源追踪与可信度

#### SRC-001 来源类型

系统至少支持以下来源类型：

- `system`
- `plugin`
- `git_repository`
- `marketplace`
- `local_import`
- `manual`
- `platform_scan`
- `central_library`
- `unknown`

#### SRC-002 证据记录

每个来源判断必须保存结构化证据，例如：目录规则、插件 manifest、Git remote、commit、安装日志、市场记录、文件内作者信息或用户确认。

#### SRC-003 可信度分数

可信度必须是 0–100 的整数。分数必须由可测试的确定性规则计算；MiniMax 可补充候选信息，但不得单独决定最终分数。界面必须同时显示分数和主要判断依据。

#### SRC-004 已确认与推断

来源必须区分：

- 已确认：有应用安装记录、用户确认或强证据唯一匹配
- 推断：根据多项证据得出的高、中、低可信度判断
- 未知：证据不足

#### SRC-005 用户纠正

用户必须能纠正来源。人工确认优先于后续自动推断，但系统仍需保留原始证据和修改历史。

### 6.4 中央 Skill 库

#### LIB-001 中央主副本

每个纳管 Skill 必须在应用工作区拥有唯一主目录。中央库中的 Skill 使用稳定内部 ID 标识，显示名称或 slug 修改不得破坏历史和映射关系。

#### LIB-002 注册与纳管

扫描到的外部实例默认只登记，不自动移动。用户选择纳管时，系统必须：

1. 校验来源目录；
2. 创建中央库副本；
3. 计算基线哈希；
4. 建立外部实例关联；
5. 创建初始快照；
6. 记录操作历史。

#### LIB-003 快照

中央 Skill 必须支持手动快照和系统恢复点。编辑、覆盖发布、导入更新和永久删除前必须按技术方案创建适当恢复点。

#### LIB-004 内容完整性

中央库必须保留 Skill 目录内除明确排除项外的全部文件。排除项必须可配置且默认只包括版本控制内部数据和临时文件。

### 6.5 多 Agent 映射与同步

#### MAP-001 一对多映射

同一个中央 Skill 必须能映射到一个或多个 Agent。每个映射必须记录目标平台、目标路径、同步模式、已发布快照、内容哈希和最近状态。

#### MAP-002 同步模式

应用必须支持复制模式；在操作系统和目标平台支持时可提供符号链接模式。默认模式为复制。选择符号链接前必须显示跨平台兼容性提示。

#### MAP-003 选择性发布

用户必须能选择一个或多个 Agent 发布指定快照，也能从单个 Agent 移除映射而保留中央主副本。

#### MAP-004 漂移检测

目标 Agent 中的受管副本被外部修改时，应用必须标记漂移，不得静默覆盖。用户可选择导入变更、覆盖目标或取消。

#### MAP-005 原子性和回滚

单个目标的发布必须使用临时目录和原子替换策略；失败时恢复原目标。多个目标发布允许部分成功，但必须逐目标报告并可重试。

#### MAP-006 平台中心真实管理

平台中心不得以静态卡片冒充已接入功能。每个首要 Agent 必须显示真实检测状态、启用状态、Skill 目录、默认同步模式、符号链接能力、受管 Skill 数量、最近同步结果和错误状态。

用户必须能测试路径、选择目录、保存配置、启用或停用平台，并进入该平台的受管 Skill 列表。修改平台目录或停用仍有映射的平台前，必须展示影响范围并禁止静默遗弃受管目标。

### 6.6 安装与导入

#### IMP-001 支持来源

第一代产品必须支持：

- 本地目录
- GitHub/Git 仓库及可选子目录、分支或 commit
- ZIP 压缩包
- 上游已经支持的市场来源

#### IMP-002 安装预览

真正写入中央库前，应用必须显示发现的 Skill、目标名称、来源、包含的脚本、文件数量、冲突和计划动作。

#### IMP-003 外部内容不执行

导入、扫描、摘要和预览过程中不得执行 Skill 自带脚本、安装命令或 Hook。

#### IMP-004 安装记录

成功或失败的导入必须保存来源 URL/路径、commit、子目录、时间、目标 Skill、状态和错误摘要。敏感请求头和 API Key 不得入库。

#### IMP-005 冲突策略

名称或目标路径冲突时，用户必须选择取消、作为新 Skill 导入或更新现有 Skill。更新现有 Skill 前必须创建恢复点。

### 6.7 编辑

#### EDT-001 编辑范围

应用必须支持编辑中央 Skill 中的文本文件，至少包括 Markdown、YAML、JSON、TOML 和纯文本。二进制文件只提供预览或外部打开。

#### EDT-002 保存安全

写入必须先写临时文件，再原子替换。保存前校验目标仍在中央 Skill 根目录内，禁止路径穿越。

#### EDT-003 编辑历史

首次保存前创建恢复点；连续编辑可按会话合并，但用户必须可以手动创建快照并查看差异。

#### EDT-004 外部编辑器

用户可打开文件或目录到系统默认编辑器。返回应用后必须检测外部变更。

#### EDT-005 编辑器用户闭环

中央 Skill 详情必须提供真实编辑模式，而不是只暴露后端保存接口。编辑模式至少包括文件选择、文本编辑、格式类型、未保存状态、保存、放弃修改、保存中状态、格式错误、保存结果和恢复点信息。

外部扫描实例默认保持只读；用户尝试编辑时必须先提示纳入中央库。切换文件、关闭编辑器、切换路由或退出应用时，如存在未保存内容，必须二次确认。保存成功后必须刷新内容哈希、快照、差异和映射过期状态；保存失败时保留用户尚未保存的文本。

### 6.8 删除、回收站与恢复

#### TRS-001 默认软删除

中央 Skill 的“删除”默认移动到应用回收站，不得立即永久删除。数据库记录必须保留原路径、删除时间、关联映射和来源。

#### TRS-002 删除计划

删除前必须显示受影响的中央文件、Agent 映射和外部实例。用户可以选择只移除某个映射、移入回收站或取消。

#### TRS-003 恢复

用户必须能将回收站项目恢复到原位置。原位置冲突时必须提供取消、恢复为新名称或由用户选择目标，禁止静默覆盖。

#### TRS-004 永久删除

永久删除只能从回收站触发，必须二次确认。第一代产品不自动清空回收站。

#### TRS-005 路径安全

任何递归移动或删除前必须解析绝对路径，并确认目标位于应用中央库、应用回收站或用户明确选择且在本次操作预览中的目录内。

### 6.9 AI 信息采集与提炼

#### AI-001 任务路由

默认职责如下：

| 任务 | 默认执行方 |
|---|---|
| 文件、路径、Git、YAML、manifest、哈希 | 本地确定性代码 |
| 章节提取、标签候选、用法要点和结构化信息采集 | MiniMax API |
| 一句话简介、内容提炼、冲突解释和需要判断的结果 | OpenAI GPT-5.6 |

模型和职责必须可配置。系统不得把本地确定性采集错误地标记为 AI 生成。

#### AI-002 手动或按需调用

首次扫描不得无条件向模型发送全部 Skill。用户可在设置中启用自动补充，也可在详情页按需生成。默认策略为按需生成并缓存结果。

#### AI-003 结果溯源

每个 AI 结果必须保存并展示：服务商、模型 ID、模型显示名、任务类型、提示词版本、输入内容哈希、生成时间、状态和 Token/用量（如果提供）。

#### AI-004 原文优先

AI 简介不得替代原始 `SKILL.md`。用户必须能快速查看原文，并看到“AI 生成内容可能有误”的适度提示。

#### AI-005 缓存失效

输入文件内容哈希、模型、提示词版本或路由变化时，已有结果必须标记为过期。用户可继续查看旧结果，但系统不得将其标记为最新。

#### AI-006 降级

未配置 API Key、连接失败、配额不足或模型不可用时，扫描、查看、安装、编辑、同步和回收站功能必须继续工作。界面显示本地解析结果和明确错误状态。

#### AI-007 数据最小化

发送给模型的内容必须限定为当前任务所需文件。调用前必须排除已识别的密钥、凭据文件和用户配置的敏感路径。调用日志不得保存完整敏感请求。

#### AI-008 生成、取消与重新生成入口

Skill 详情必须提供“生成简介/用法”和“重新生成”入口，并展示 `idle`、`running`、`success`、`error`、`cancelled` 和 `stale` 状态。运行中的任务必须可取消；重复点击不得创建无法识别的并发任务。

生成结果必须显示实际 Provider、模型 ID、职责、生成时间、缓存或过期状态，以及适度的 AI 可能出错提示。默认仍为按需生成；设置中可提供“对缺少或过期摘要的中央 Skill 自动补充”开关，该开关默认关闭并说明会发送最小必要内容及可能产生 API 费用。

### 6.10 模型与 API 配置

#### CFG-001 配置入口

设置中必须提供“模型与 API”页面，分别配置 MiniMax 和 OpenAI：API 地址、API Key、默认模型、任务职责、超时和连接测试。

#### CFG-002 密钥存储

API Key 必须保存到操作系统安全凭据服务：

- Windows Credential Manager
- macOS Keychain
- Linux Secret Service；不可用时禁止静默退回明文存储

SQLite 和普通设置文件只保存凭据引用及脱敏尾号。

#### CFG-003 模型显示

应用必须在设置页和 AI 结果处显示实际调用的模型，而不只显示营销名称。例如：服务商 `OpenAI`、模型 ID `gpt-5.6`、职责“内容提炼与决策”。

#### CFG-004 连接测试

连接测试必须区分：凭据缺失、鉴权失败、网络失败、模型不存在、配额/限流和未知错误。错误信息不得包含完整 API Key。

### 6.11 操作历史与审计

#### AUD-001 操作日志

系统必须记录安装、导入、纳管、编辑保存、快照、发布、移除映射、回收、恢复、永久删除、来源纠正和 AI 生成事件。

#### AUD-002 可读性

历史记录至少包含操作类型、目标、时间、结果和错误摘要。涉及文件变化时应关联前后哈希或快照。

#### AUD-003 隐私

日志不得记录 API Key、认证头、完整环境变量值或被识别为秘密的文本。

### 6.12 UI 与品牌

#### UI-001 产品品牌

产品名称为 Skill Studio Pro。必须使用独立 Logo、图标、包标识和更新地址，同时在关于页注明基于开源 Skill Studio 改造。

#### UI-002 视觉语言

默认主题为深海蓝暗色，借鉴 macOS 27 Liquid Glass 的层次、半透明、柔和高光、圆角和动效。不得直接复制仅授权 Apple 平台使用的设计资源。

#### UI-003 信息密度

桌面窗口在 1280×800 时必须完整使用；最小支持 900×600。窗口可缩放，主要功能不得只能通过悬停访问。

#### UI-004 可访问性

文字与背景对比度应满足 WCAG 2.2 AA；支持键盘导航、清晰焦点、减少动态效果设置和非颜色唯一状态表达。

#### UI-005 模型状态可见

模型与 API 页面必须展示每个 Provider 的连接状态、当前模型和职责。Skill 详情中的 AI 内容必须展示模型归属。

#### UI-006 真实桌面关键流程验收

结构测试和 CSS 断言不能替代真实桌面验收。发布候选必须在真实 WebView 中验证 900×600、1280×800 和高 DPI 下的扫描、详情、编辑、AI 生成、平台配置、发布、回收站和恢复流程，并保存书面结果；关键操作不得被遮挡、只能悬停访问或因缩放无法完成。

### 6.13 开源与许可证

#### OSS-001 开源发布

项目源代码必须以公开仓库形式发布，默认采用 Apache-2.0。

#### OSS-002 上游归属

必须保留上游 LICENSE、NOTICE 和适用版权声明；README 和关于页必须说明上游来源、基线 commit 和主要修改。

#### OSS-003 第三方资产

发布前必须生成第三方依赖与资产清单。无法确认再分发许可的 Logo、截图或品牌资产不得进入发布包。

#### OSS-004 秘密与用户数据

仓库必须忽略 API Key、本地数据库、日志、Skill 内容、回收站和用户配置。CI 必须执行秘密扫描。

#### OSS-005 独立公开仓库与发布流水线

Skill Studio Pro 必须拥有独立的公开 `origin` 仓库、默认 `main` 分支、可复现构建说明和自己的 Release 地址。不得把 Pro 发行包发布到上游仓库，也不得恢复使用上游自动更新密钥或端点。

公开仓库的 Pull Request 或主分支 CI 必须在 Windows、macOS、Linux 真实 Runner 上执行规定门槛。首个公开 Beta 必须附版本号、变更说明、已知限制、SHA-256 校验文件和至少一个已验证安装包。

#### OSS-006 第三方依赖与安全文档

公开发布前必须提供 `SECURITY.md`、`CONTRIBUTING.md`、第三方依赖/资产清单和可机器读取的 SBOM。无法确认许可证兼容性的依赖或资产不得进入发布包。签名、公证或 Linux 包签名若尚未完成，必须在 Beta 已知限制中明确，不得宣称已签名。

## 7. 非功能需求

### NFR-001 跨平台

核心功能必须在当前受支持的 Windows、macOS、主流桌面 Linux 发行版上工作。平台差异通过后端适配层处理，业务层不得散布硬编码路径分隔符。

### NFR-002 性能

在 1,000 个 Skill、总计 100,000 个文件的测试数据下：

- 冷启动在普通 SSD 设备上应在 5 秒内显示已有索引；
- 首次扫描应持续显示进度且可取消；
- 已有索引搜索的 P95 响应时间应低于 150 毫秒；
- 单 Skill 详情打开的 P95 应低于 300 毫秒，不含外部 AI 调用。

### NFR-003 稳定性

单个 Skill 解析失败不得中止整次扫描。数据库迁移、发布、恢复和导入必须可检测失败并提供恢复或重试路径。

### NFR-004 安全

所有外部路径必须规范化并防止路径穿越。网络请求必须使用 TLS。外部 Skill 内容在用户明确操作前不得执行。

### NFR-005 隐私

默认不上传 Skill 内容。只有用户触发或明确启用 AI 任务时，才将最小必要内容发送给所选 Provider。

### NFR-006 可维护性

新增 Agent 必须通过 Adapter/平台定义扩展，不应修改中央库核心流程。新增模型 Provider 必须通过统一接口接入。

### NFR-007 可测试性

Rust 业务逻辑必须可在不启动桌面窗口时测试；前端通过 Mock IPC 测试；外部 API 必须可由本地 Mock Server 替代。

## 8. 错误状态

系统至少区分并可展示：

- 扫描根不存在
- 权限不足
- `SKILL.md` 解析失败
- 文件编码异常
- 数据库不可用或迁移失败
- 来源证据冲突
- 目标映射漂移
- 安装来源不可访问
- Git/ZIP 导入失败
- 写入或原子替换失败
- 回收站恢复冲突
- 凭据存储不可用
- API 鉴权、网络、限流、配额和模型错误
- 操作取消

错误必须包含可操作的下一步，不得只显示“未知错误”。

## 9. 第一代发布门槛

满足以下条件方可发布：

1. 本文件全部 Must 需求有自动化测试或书面人工验收记录。
2. 五个首要 Agent 的扫描、映射和移除流程通过跨平台契约测试。
3. 删除默认进入回收站，永久删除路径安全测试通过。
4. API Key 不出现在数据库、日志、崩溃输出和 Git 历史中。
5. MiniMax 与 OpenAI 调用可被禁用和 Mock，离线核心功能可用。
6. Windows、macOS、Linux 构建成功。
7. LICENSE、NOTICE、README 上游归属和第三方清单完整。
8. UI 在 900×600、1280×800 和高 DPI 环境通过关键流程验收。
9. 中央 Skill 可通过真实 UI 完成编辑、保存、恢复点、差异和未发布状态闭环。
10. Skill 详情可真实触发、取消和重新生成 AI 简介/用法，并展示实际模型归属。
11. 平台中心和扫描根管理已接入真实 IPC，不再以静态卡片作为 P0 功能。
12. 在隔离目录完成至少一次真实 Windows 全生命周期验收，且不触碰用户真实 Agent 目录。
13. 1,000 Skill/100,000 文件发布候选基准实际执行并记录；若不达标，必须有阻断结论或批准的限期豁免。
14. 独立公开仓库已运行三平台 CI；Windows、macOS、Linux 的构建和启动结果有可追溯记录。

## 10. 需求变更规则

对中央库数据模型、删除语义、AI 数据发送范围、许可证、首要平台列表或公开 API 的修改，必须同时更新本文件、PRD、技术方案和自动化测试文档。实现代码不得以“技术方便”为由静默改变本文件定义的用户行为。
